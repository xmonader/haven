Privacy and secrets,
built into version control.

Haven is a from-scratch VCS for code that isn't ready for the public world yet. Private branches, need-to-know sharing, and encrypted credentials are first-class — enforced by storage and cryptography, not by discipline.

The problem

Mainstream VCS gets two things wrong

…and you pay for both, every day. Haven makes them structural instead of procedural.

📌

Privacy is a sticky note

Every branch is equal; keeping one private means remembering not to push it. One slip and your half-baked work is public.

🔓

Secrets are plaintext

A .env in the working tree becomes a plaintext blob in every clone, backup, and push — forever in history.

🏢

Access lives in a vendor

Who can read what is stored in a hosting vendor's database. It doesn't travel with the repo and isn't verifiable offline.

The model

Confidentiality is two independent axes

Access — who's allowed? (public · restricted · haven), enforced by the server. Secret — is it ciphertext on disk? enforced by crypto. They compose, tied by one rule: a secret's decryption recipients are exactly the people who can read its branch.

What's inside

A complete VCS — with the hard parts done right

Local history, branching and merging, remotes, a signed access policy, and encrypted secrets. Decentralized like git: every clone is whole, everything works offline.

📚 Local & history

add commit status log diff reset restore tag · cherry-pick revert rebase stash bisect. Symlinks tracked.

🌿 Branches & havens

branch · haven (private) · publish (one-way graduation) · three-way, rename-aware merge with conflict markers.

🛰️ Remotes

serve (HTTP/S) · push (refuses havens) · fetch · pull · clone · sync (carries havens between your machines).

🔑 Identity & access

A portable, Ed25519-signed policy chain is the authorization root. Grants are verifiable offline; the server enforces every fetch and ref update.

🔐 Secrets

secret add/ref/rotate/status. Files matching a mark are encrypted to the ref's readers on commit; rotate re-keys after a membership change. The store holds only ciphertext.

🛡️ Hardening

fsck · gc (grace period) · repack (delta compression). Bodies capped, requests signed with replay protection, schema-versioned DB. Tested under -race.

Quick start

Sixty seconds to a private, encrypted repo

One static Go binary, no daemon, no cgo. Build with make build, then:

# new repo (seeds default secret marks: .env, *.pem, …)
hv init
hv key gen                       # your encryption identity; you join the members
echo "DATABASE_URL=…" > .env
hv add . && hv commit -m "app"   # .env is encrypted automatically

# private, unready work — push will refuse to leak it
hv haven create scratch
hv publish scratch --as feature  # graduate to a public branch when ready

# share with a team over TLS; pin the policy root
hv remote add origin <url> --kind team
hv push origin main              # the secret travels as ciphertext
hv serve --kind team --policy-root <root-key> --tls-cert cert.pem --tls-key key.pem

Security & status

Honest about where it stands

A working VCS, hardened — not yet externally audited

Haven enforces confidentiality with cryptography and access with a signed policy the server can enforce but not forge. The crypto/data path is hardened — private secret-file permissions, decrypted-content integrity checks, atomic writes, request signing with replay protection, bounded resource use — and tested under the race detector.

It has not had an external security audit. Suitable for solo and small-trusted-team use over TLS; get an audit before relying on it as the sole control for high-value secrets in a hostile environment.
